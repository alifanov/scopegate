## Recurring Issues

Нет повторяющихся (2+ аудита подряд) активных уязвимостей. Напротив — основные находки прошлых аудитов закрыты в коде:

- **SSRF в скачивании image/video** (был high) — **исправлено**: всё идёт через `safeFetch`, который валидирует все A/AAAA-записи, пинит IP (защита от DNS-rebinding/TOCTOU), только `https:`, ре-валидация на каждом редиректе.
- **GAQL-инъекция в Google Ads** (был high) — **исправлено**: все интерполируемые значения валидируются (`gaqlNumericId` `^\d+$`, `gaqlDate` `^\d{4}-\d{2}-\d{2}$`, enum-схемы, `maxResults: z.number()`).
- **Неподписанный OAuth state** (был high) — **исправлено**: HMAC-SHA256 подпись + `timingSafeEqual`, CSRF-cookie сверяется отдельно (`oauth-state.ts`, `oauth-flow.ts`).
- **Слабая генерация ключа шифрования** — `crypto.ts` использует AES-256-GCM со случайным IV на каждый вызов, PBKDF2 100k.

## Hypotheses

- **#112 (cuid → крипто-стойкие API-ключи)** — судя по коду, **уже исправлено**: `generateMcpApiKey()` использует `randomBytes(32).toString("base64url")` (256 бит), и оба места создания эндпоинтов (`endpoints/route.ts:88`, `regenerate-key/route.ts:32`) вызывают именно его. Issue остаётся открытым — вероятно фикс уже залит, нужно верифицировать и закрыть. Не переоткрывал.
- **API-ключ в URL-пути** MCP-эндпоинта (`/api/mcp/[apiKey]`) — архитектурный выбор MCP; код избегает логирования ключа в OTel (нормализация маршрута). Риск утечки через access-логи прокси/referrer остаётся, но это design-level, не код-уязвимость. Наблюдать.
- **CSP всё ещё в Report-Only** (`next.config.ts`) — by design; не находка. Промоут до enforce — отдельная задача.
- **csp-report** — неаутентифицированный POST-сток; тело обрезается (`slice(0, 256)`), амплификации/инъекции лога не видно. Низкий риск.

## Recommendations

1. **Потоковый лимит размера при скачивании медиа** (medium → issue #144):
   найдено: `image-utils.ts` и `youtube.ts` буферизуют весь ответ до проверки лимита → OOM-DoS общего процесса любым тенантом.
   фикс: общий `readBodyWithLimit(res, maxBytes)` с обрывом чтения по достижении лимита; отклонять ответы без/с превышающим `Content-Length`.
   критерий: тело без `Content-Length` сверх лимита отклоняется без исчерпания памяти; есть unit-тест.

2. **Доверенный источник IP для rate-limit** (low, не filed):
   найдено: `getClientIp` доверяет левому `X-Forwarded-For`.
   фикс: за известным прокси брать правый элемент XFF / фиксированный hop count или `cf-connecting-ip`/`x-real-ip`.
   критерий: подделка `X-Forwarded-For` не сбрасывает счётчик брутфорс-лимита. (Низкий приоритет — ключи 256-битные, брутфорс нереалистичен.)

3. **Constant-time сравнение `CRON_SECRET`** (low, не filed):
   найдено: `authHeader !== expected`.
   фикс: `crypto.timingSafeEqual`.
   критерий: сравнение секрета не зависит от времени по длине совпадения.

4. **Не возвращать `error.message` клиенту** (low, не filed) в `accept-invite`/`admin/users` — отдавать обобщённый текст, детали только в лог.

5. **Верифицировать и закрыть #112** — фикс уже в коде.
