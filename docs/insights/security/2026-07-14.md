## Hypotheses

- **Инвайт-гонка (low).** Пока impact мал (нет автоприсвоения проекта), но если появится инвайт с предзаданным членством/ролью — гонка станет привилегированной. Сигнал к переводу в medium: любой флоу, где accept-invite сразу выдаёт роль/членство. Fix дешёвый заранее: `updateMany({ where: { token, usedAt: null } })` + проверка `count === 1`.
- **CSP пока Report-Only** (`Content-Security-Policy-Report-Only`, `script-src 'self' 'unsafe-inline'`). Не находка — сознательное решение (промежуточный этап). Отслеживать: когда отчёты чистые, продвинуть до enforcing `Content-Security-Policy` и убрать `'unsafe-inline'` для script-src.
- `x-powered-by: Next.js` отдаётся на всех ответах — минимальное раскрытие версии/стека (info-only, не задача).

## Recommendations

1. **CRLF в заголовках Gmail (medium → задача #161).** Найдено: `subject` не экранирует `\r\n` в `buildRawEmail`. Фикс: вырезать/отклонять `\r`/`\n` в `subject` (и `to` — defense in depth) на уровне zod-схемы или при сборке MIME. Критерий приёмки: `buildRawEmail(..., "Hi\r\nBcc: e@evil.com", ...)` не порождает отдельного заголовка `Bcc:`; добавлен unit-тест.
2. **Атомарное погашение инвайта (low, без задачи).** Заменить read-then-update на `updateMany({ where: { token, usedAt: null } })` с проверкой `count === 1`. Критерий: два параллельных accept с одним токеном создают ≤1 аккаунт.
3. **Constant-time CSRF-сравнение (low, без задачи).** Привести `oauth-flow.ts:220` к `timingSafeEqual` для согласованности с проверкой подписи state. Критерий: сравнение CSRF-cookie использует constant-time.
