## Hypotheses

- **AuditLog рост таблицы** — INSERT 53 мс может расти по мере накопления строк; при отсутствии архивирования через 3–6 мес. INSERT может деградировать значительнее. Следить за динамикой.
- **`twitter_post_tweet` 926 мс** — близко к 1 с порогу; один из следующих снапшотов может потребовать issue.
- **4 HTTP 500 на `/api/auth/[...all]`** — малый объём (4 за 7 дней); может быть race при конкурентном refresh. Если появится в следующих снапшотах — требует расследования.

## Recommendations

1. **Issue #140 (high)** — Расследовать 34 MCP 500: запросить трейсы с filter `http.status_code=500 AND http.route="/api/mcp/[apiKey]"` в SigNoz; сгруппировать по `error.message`.
2. **Issue #122 (high, обновлено)** — threads_publish_thread регрессировал: 11.9 с → 18.6 с. Добавить early-return с `partial_success` при превышении 20 с вместо ожидания до 25 с.
3. **Issue #141 (medium)** — LinkedIn p99 2 с: проверить на N+1 вызовы, добавить таймаут с понятной ошибкой.
4. **Threads OAuth callback 1.5 с** — задержка внешняя (graph.threads.net token exchange ~942 мс); оптимизировать нельзя, добавить UX-индикатор при повторном подключении.
5. **AuditLog архивирование** — запланировать retention policy (например, удалять записи старше 90 дней) до того как таблица начнёт замедлять INSERT.
