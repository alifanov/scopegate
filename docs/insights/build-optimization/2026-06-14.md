# Build Optimization — 2026-06-14

**Scope:** `Dockerfile`, `docker-entrypoint.sh`, `docker-compose.yml`, `.dockerignore`, `next.config.ts`, `package.json` scripts, `pnpm-lock.yaml`, `Makefile`, `prisma.config.ts`, `docker/prisma.config.ts`, `tsconfig.json`, `vitest.config.ts`. No `.github/workflows/` (deploy is Coolify building the Dockerfile on push to `master`).

## Pipeline overview

`git push master` → Coolify clones repo → `docker build` (multi-stage `node:22-slim`: **deps** copies manifests+prisma → `pnpm install --frozen-lockfile` with BuildKit pnpm store cache mount → **builder** copies `node_modules` + `COPY . .` → `prisma generate && next build` (standalone output) → **runner** copies `.next/standalone` (71 MB) + `.next/static` (1.5 MB) + `public` + `prisma`, installs prisma CLI separately into `/prisma-runtime`) → container start runs `prisma migrate deploy` then `node server.js`. Both prior-audit issues are resolved: **#91** (multi-stage + standalone) and **#92** (BuildKit pnpm cache) are closed and present in the Dockerfile. Runtime image ≈ standalone (71 MB) + prisma CLI (~25–40 MB) + base. Remaining waste is in the **build trigger**, not the image: docs/config-only commits bust the `COPY . .` layer, and there is no CI gate to stop broken commits reaching Coolify.

## Recurring

No recurring items. Prior audit (2026-06-07) filed #91 (multi-stage+standalone) and #92 (BuildKit pnpm cache) — both now **closed/implemented**. The "no CI" note from 2026-06-07 is elevated to a filed issue (#120) this round because the history now shows concrete broken-build incidents reaching `master`.

## Recommendations

- `.dockerignore` → add `*.md`, `.darkflow`, `skills-lock.json`, `components.json`, `.mcp.json`, `**/__tests__`, `*.test.ts` (do NOT exclude `prisma/migrations` — needed by runner) → acceptance: a CLAUDE.md/CHANGELOG/.darkflow-only commit shows `CACHED` on the `next build` layer in the Coolify build log (#119).
- `.github/workflows/ci.yml` → on push to `master` + PR: pnpm install `--frozen-lockfile` → `prisma generate` → `next build` (not `pnpm build`, to skip `migrate deploy`) → `pnpm test` → `pnpm lint`, with pnpm store cache → acceptance: lockfile desync / `next build` failure fails CI before the Coolify deploy completes (#120).

## Notes (minor / not filed)

- **Duplicate Prisma CLI in runner (~25–40 MB)**: the runner does a second `npm install prisma@X` into `/prisma-runtime` purely for `migrate deploy` at startup, ~doubling the standalone image. This is largely architecturally necessary (the standalone bundle has no CLI, and the DB isn't available at build time). Savings are modest vs the effort/risk of an alternative migration mechanism (init job / one-off container) — left as a note.
- The install-cache layer ordering is already correct (manifests + prisma copied before `pnpm install`); the npm install of the prisma CLI is gated on `COPY package.json /tmp` so it stays cached across builds unless `package.json` changes.
- `docs/` and `.darkflow.d/` are already in `.dockerignore`, so observability/security snapshot commits correctly do not bust the build cache.
