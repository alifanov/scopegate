## Update (задача #172, закрыта)

Причина найдена: orphaned `GET`-спан — это **не отдельное соединение**, а те же самые MCP SSE-подключения (`GET /api/mcp/[apiKey]`), посчитанные дважды.

`@opentelemetry/instrumentation-http` называет корневой SERVER-спан только по HTTP-методу (`GET`/`POST`/…), а `http.route` кладёт отдельным атрибутом, не переименовывая сам спан. Next.js создаёт свой внутренний дочерний спан с полным именем (`GET /api/mcp/[apiKey]`) — а `slow_endpoints`-запрос в check-signoz группирует p99 по полю `name`, поэтому одно и то же долгоживущее SSE-соединение попадало в две разные корзины: `GET /api/mcp/[apiKey]` (~63.9 мин) и голый `GET` (~60.4 мин). Совпадение диапазонов 3 снапшота подряд объясняется тем, что это буквально одна и та же популяция запросов.

Проверено вживую через SigNoz API (`traces` list с `orderBy durationNano desc`): среди спанов с `name=GET AND http.route=""` максимальная длительность за 24ч — ~1.2 сек, а не 60 минут — подтверждает, что "orphaned"-корзина сама по себе безобидна; весь вклад в p99 шёл от корневых спанов реальных MCP-подключений.

**Фикс:** `src/instrumentation.node.ts` — `requestHook` для `instrumentation-http` теперь вызывает `span.updateName(`${method} ${route}`)` в дополнение к `setAttribute("http.route", ...)`, как уже делал undici-хук для исходящих запросов. Корневой спан теперь называется `GET /api/mcp/[apiKey]` вместо голого `GET` — двойной подсчёт устранён для всех маршрутов, не только MCP.

Ожидаемый эффект в следующем снапшоте (07-17): отдельная "orphaned `GET`" строка в таблице латентности исчезнет — весь трафик схлопнется в уже существующие route-именованные корзины.
