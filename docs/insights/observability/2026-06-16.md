## Hypotheses

- **Unnamed `GET` span with p99 ~17 min** — likely an untagged SSE or WebSocket-like span (possibly browser or proxy). Not creating an issue yet; need to confirm it's a keep-alive artifact and not a real hanging request. Will watch next snapshot.
- **AuditLog SELECT p99 51 ms** — under threshold, but trending upward as audit log table grows. May need a composite index on `(endpointId, createdAt)` once table exceeds ~1M rows.
- **LinkedIn p99 1.7 s** — LinkedIn's API is inherently slow for POST operations. No action needed unless users report timeouts.

## Recommendations

1. **#108 (CRITICAL)** — Sanitize Threads API URLs in `service-fetch.ts` before using as span names. Strip `?access_token=` query params. Unblock immediately.
2. **#109 (HIGH)** — Add graceful handling for persistent Meta 400 errors: after N consecutive failures, auto-revoke connection and notify user (similar to `OAuthTokenError` flow).
3. **#124 (MEDIUM, auto-approved)** — Add `@@index([projectId])` to `McpEndpoint`; create migration. Will cut SELECT p99 from 163 ms to < 50 ms.
4. **#122 (MEDIUM)** — Investigate `threads_publish_thread` timeout risk; consider splitting into create + publish with separate OTel spans to identify which step is slow.
5. **#111 (MEDIUM)** — Profile Google OAuth callback: token exchange latency vs. DB write vs. redirect. Cache or parallelize where possible.
